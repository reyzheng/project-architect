## 4. Subagent Routing Requirements

Claude Code relies heavily on each subagent frontmatter `description` for automatic delegation.
Descriptions must be explicit and non-overlapping.

Use these exact recommended descriptions in generated subagent templates:

```yaml
plan-agent: Creates architecture documents, requirements specifications, threat models, and software design workflows.
build-agent: Defines project build, compilation, packaging, artifact generation, and dependency tree workflows.
test-agent: Defines unit test, integration test, SAST, DAST, SCA, secret scanning, and code review workflows.
release-agent: Defines release, deployment, smoke test, rollout, rollback, and delivery verification workflows.
```

Avoid generic repeated phrases such as `analyze codebase` or `generate workflows` in multiple subagent descriptions.

## 6. Claude Workflow Rules

Generated workflow JS files must follow these rules:

- workflow JS is an orchestrator only
- workflow JS must not directly read/write files or execute shell commands
- workflow JS must use `agent()` for filesystem work, shell work, Python execution, CLI/API execution, and report writing
- workflow execution must not depend on stopping mid-run to ask the user a question
- if user confirmation is needed, generate a draft and summary, then end; the user can invoke `pipeline-architect` again to revise or confirm

## 7. Config Validation Requirement

For workflows that need `pipeline_config.local.yaml`, the first stage must validate config through `agent()`.

The validation agent must:

- read `.claude/pipelines/{flow}/pipeline_config.local.yaml`
- check required keys
- check required local paths when possible
- return only redacted status through a schema
- never return tokens, passwords, API keys, private keys, or secret values to workflow JS

Later execution agents or Python scripts should read `pipeline_config.local.yaml` themselves.
Do not pass secrets through workflow variables or prompts.

## 8. Failure Handling Requirement

Generated workflows must include explicit failure handling.

Each stage result should include, when possible:

- `success`
- `stage`
- `exitCode`
- `stdoutSummary`
- `stderrSummary`
- `logPath`

If a stage fails:

- stop dependent stages
- call a reporter agent
- create `.claude/pipelines/{flow}/reports/error-report.md`
- redact secrets
- include failed stage, script path, exit code, stdout/stderr summary, local log path, suggested fix, and suggested rerun command
